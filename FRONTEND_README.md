## 1. Project Overview & Core Rule

BacPrep Hub is a React single-page application that serves as the student-facing and admin-facing interface for the platform. It communicates exclusively with the BacPrep Hub backend REST API. The frontend owns zero business logic — it renders, routes, collects input, and displays what the backend returns.

**The single most important rule of this entire frontend:** The frontend never calls Cloudinary, Gemini, or Chargily directly. Every external service is accessed exclusively through the backend API. The only external redirect that originates from the frontend is sending the student to Chargily's hosted payment page — and even then, the URL for that redirect comes from the backend. The frontend never constructs Chargily URLs on its own.

The backend is hosted on Render. The frontend is hosted on Vercel. They are completely separate deployments. The frontend knows about the backend only through the environment variable VITE_API_URL. These two services communicate only over HTTP.

## 2. Tech Stack

Vite is the build tool and development server. It is chosen over Create React App because it is significantly faster, uses native ES modules, and provides instant hot module replacement. The framework preset in Vite is React.

React 18 is the UI library. The entire application is built with functional components and hooks. No class components anywhere.

React Router version 6 handles all client-side navigation. It manages the route definitions, the protected route wrappers, and the admin route wrappers. The useNavigate and useLocation hooks from React Router are used throughout for programmatic navigation and reading location state.

Tailwind CSS is the styling approach. There are no custom CSS files in this project. Every visual style is applied through Tailwind utility classes directly in the JSX. The tailwind.config.js file extends the default theme with the project's specific color palette if needed.

Shadcn/ui provides the pre-built component library. It is built on top of Radix UI primitives, which means every component is accessible out of the box. The components that will be used heavily are: Button, Input, Dialog, AlertDialog, Toast, Card, Select, DropdownMenu, Badge, Skeleton, and Separator. Shadcn components live in the src/components/ui folder and are generated by the Shadcn CLI — they are never edited manually after generation.

Axios is the HTTP client. A single configured Axios instance is created once and imported everywhere. No component ever creates its own Axios instance or uses the native fetch API. The reasons for this are explained in detail in section 5.

Recharts is the charting library. It is used for one specific purpose: the bar chart on the student dashboard that displays average scores per chapter. It is lightweight, built specifically for React, and integrates cleanly with Tailwind colors.

React Context is used for global state management. There is no external state management library such as Redux, Zustand, or Jotai. The application's global state is limited to authentication data — user, token, and credit balance — which is simple enough to manage with a single Context and the useReducer or useState hooks.

## 3. Folder Structure

The entire frontend lives in a folder called client at the root of the repository, alongside the server folder. The structure inside client is as follows.

The src folder contains everything. Inside src there are six subfolders and two root files.

The api folder contains exactly one file: axios.js. This file creates and exports the configured Axios instance. Every API call in the entire application imports from this single file. Nothing else lives in this folder.

The components folder has two layers. The ui subfolder is entirely managed by the Shadcn CLI and is never touched manually. The rest of the components folder contains custom shared components used across multiple pages: the Navbar, the ProtectedRoute wrapper, the AdminRoute wrapper, and the ChatbotWidget. More on each of these in section 10.

The context folder contains one file: AuthContext.jsx. This file defines the AuthContext, the AuthProvider component that wraps the entire app, and the initial state. The useAuth custom hook is placed in the hooks folder rather than here to keep the two concerns separate.

The hooks folder contains one file: useAuth.js. This hook is a thin wrapper around useContext(AuthContext) with a guard that throws a descriptive error if it is called outside the AuthProvider. All components import useAuth from here rather than importing AuthContext directly.

The pages folder contains one file per page of the application. The admin pages live in a subfolder called admin to keep them separate from student pages. The full list of page files is: Landing.jsx, Login.jsx, Register.jsx, Dashboard.jsx, ExamLibrary.jsx, QuizSelect.jsx, QuizActive.jsx, QuizResults.jsx, Credits.jsx, PaymentSuccess.jsx, PaymentFail.jsx, and inside the admin subfolder: AdminDashboard.jsx, AdminExams.jsx, AdminQuestions.jsx, and AdminCredits.jsx.

The utils folder contains one file: formatters.js. This file exports pure helper functions for displaying data in a human-readable format. It includes a function to format a date as a readable Algerian-style string, a function to display a percentage rounded to one decimal place, and a function to display a credit balance with the word "crédits" appended.

App.jsx is the top-level component that contains the router definition only. It imports all page components and defines the route tree. It contains no UI of its own.

main.jsx is the application entry point. It renders the App component wrapped in the AuthProvider. Nothing else should be added at this level.

At the root of the client folder, outside of src, there are four important files. The .env file holds environment variables. The .env.example file documents all variable names without values and is the only one committed to version control. The vercel.json file contains the SPA rewrite rule required for Vercel deployment, explained in section 15. The vite.config.js file holds the Vite configuration.

## 4. Environment Variables

There are exactly two environment variables for the frontend. Both must be prefixed with VITE_ to be accessible in the browser bundle — Vite strips any environment variable without this prefix from the client-side code at build time.

VITE_API_URL is the base URL of the backend API. In development this is http://localhost:5000. In production it is the full Render deployment URL. This variable is used in exactly one place: the Axios instance configuration in src/api/axios.js. It must never be referenced anywhere else in the codebase. If a component needs to call the API, it does so through the Axios instance, not by constructing URLs manually.

VITE_APP_NAME is optional and defaults to BacPrep Hub if not set. It is used in the HTML document title and any page-level meta tags.

The .env file is added to .gitignore before the first commit. The production value of VITE_API_URL is set in Vercel's environment variables settings panel, not in any file that gets committed.

## 5. Axios Instance & HTTP Strategy

The Axios instance is created in src/api/axios.js and is the most important infrastructure decision in the frontend. Here is everything it does.

It sets the baseURL to import.meta.env.VITE_API_URL so that every request automatically targets the correct backend regardless of environment. A component that wants to fetch the exam list calls the Axios instance with the path /api/exams — it never constructs the full URL itself.

It attaches a request interceptor that runs before every outgoing request. The interceptor reads the JWT token from localStorage under the key token and, if a token exists, adds it to the Authorization header in the format Bearer followed by the token string. This means no component ever has to think about authentication headers — they are attached automatically to every request.

It attaches a response interceptor that runs after every incoming response. This interceptor handles two specific HTTP status codes globally so that no individual component has to.

For 401 responses, the interceptor removes the token and user data from localStorage, resets the authentication context state, and redirects the user to the login page. This handles expired or invalidated tokens transparently. A student can never end up on a protected page with a broken token — they are sent to login automatically.

For 402 responses, which the backend sends when a student has insufficient credits, the interceptor navigates the user to the /credits page. This means that when a student tries to start a quiz or send a chatbot message without enough credits, they are automatically redirected to the purchase page without any individual component needing to handle this case. The 402 interception is the global solution — it must not be reimplemented component by component.

All other error codes, such as 400, 403, 404, and 500, are allowed to propagate normally to the calling component's catch block so the component can display the appropriate error message.

## 6. Authentication Context

The AuthContext is defined in src/context/AuthContext.jsx and provides authentication state to every component in the application. The AuthProvider component wraps the entire app in main.jsx.

The context holds the following state values. The user field contains the full user object returned by the backend — name, email, role, and creditBalance — or null if the user is not logged in. The token field contains the raw JWT string or null. The creditBalance field is kept separately from the user object even though it is also on the user object, because it needs to update frequently without triggering a full user object replacement. The isLoading field is a boolean that is true while the initial authentication check is running on app mount. The isAuthenticated field is a derived boolean that is true when user is not null — it is computed from user and never stored separately.

The context exposes the following methods. The login function accepts a token and a user object, saves the token to localStorage under the key token, saves the user object to localStorage under the key user, and updates the context state. It is called by the Login and Register pages after a successful API response. The logout function clears the token and user from localStorage, resets the context state to its initial unauthenticated values, and navigates the user to the landing page. The updateCredits function accepts a number and updates the creditBalance in context state. It is called after any API action that returns an updated credit balance — quiz start, chatbot message, and payment success.

On mount, the AuthProvider checks whether a token exists in localStorage. If it does, it calls GET /api/auth/me to validate the token and get a fresh user object. This is the mechanism that keeps a student logged in across page refreshes. If the call succeeds, the user object and credit balance are set from the response. If the call fails for any reason — expired token, deleted account, network error — the token is removed from localStorage silently and the user is treated as unauthenticated. During this initial check, isLoading is true. The ProtectedRoute component must not redirect to login while isLoading is true, because that would incorrectly redirect a valid user before the check completes.

The useAuth hook in src/hooks/useAuth.js wraps useContext(AuthContext) with a single guard: if the hook is called outside the AuthProvider, it throws an error with a descriptive message explaining that useAuth must be used inside an AuthProvider. Every component that needs authentication state imports useAuth and nothing else.

## 7. Routing & Access Control

All route definitions live in App.jsx and nowhere else. The router uses BrowserRouter as the provider.

There are three levels of route access in this application.

Public routes are accessible to anyone, authenticated or not. Login and Register have additional logic: if the user is already authenticated, these pages redirect to /dashboard immediately rather than showing the form.

Protected routes require authentication. The ProtectedRoute component checks the isAuthenticated value from the auth context. While isLoading is true, the component renders a full-page loading spinner rather than making any routing decision. Once loading is complete, if the user is not authenticated, the component redirects to /login and saves the current path in React Router's location state under the key from. After the user logs in, the login page checks for this saved path and redirects there instead of defaulting to /dashboard.

Admin routes require both authentication and the admin role. The AdminRoute component first checks authentication using the same logic as ProtectedRoute. After confirming the user is authenticated, it checks whether user.role equals admin. If the user is authenticated but not an admin, they are redirected to /dashboard with no error message — admin routes are simply invisible to non-admin users.

The full route map is as follows.

The slash route renders the Landing page and is public. The /login route renders the Login page and is public with redirect-if-authenticated logic. The /register route renders the Register page and is public with redirect-if-authenticated logic. The /exams route renders the ExamLibrary page and is public — exam browsing requires no login. The /dashboard route renders the Dashboard page and requires authentication. The /quiz route renders the QuizSelect page and requires authentication. The /quiz/:attemptId route renders the QuizActive page and requires authentication. The /quiz/:attemptId/results route renders the QuizResults page and requires authentication. The /credits route renders the Credits page and requires authentication. The /credits/success route renders the PaymentSuccess page and requires authentication. The /credits/fail route renders the PaymentFail page and requires authentication. The /admin route renders the AdminDashboard page and requires the admin role. The /admin/exams route renders the AdminExams page and requires the admin role. The /admin/questions route renders the AdminQuestions page and requires the admin role. The /admin/credits route renders the AdminCredits page and requires the admin role.

Any path that does not match a defined route renders a simple 404 page that shows a message and a link back to the home page.

## 8. Pages — Complete Reference

This section describes what every page does, what data it fetches, and any specific behavior it must implement.

**Landing page** is the marketing page for the platform. It has a hero section with the platform name, a one-sentence description of what it does, and two call-to-action buttons for Register and Login. Below the hero there is a features section listing the four main capabilities: exam library, quiz engine, dashboard, and AI chatbot. A Bac countdown widget is displayed prominently — this is a simple calculation of days remaining until June 10 of the current year, or June 10 of the following year if that date has already passed. The page ends with a final call-to-action section. This page makes no API calls.

**Login page** contains a form with two fields: email and password. On submit it calls POST /api/auth/login. On success it calls the login function from auth context with the returned token and user, then navigates to the path saved in location state if one exists, or to /dashboard otherwise. On failure it displays the error message returned by the backend under the form. If the user is already authenticated, this page redirects to /dashboard immediately on mount.

**Register page** contains a form with three fields: name, email, and password. Client-side validation requires the name to be at least two characters, the email to be a valid format, and the password to be at least eight characters. On submit it calls POST /api/auth/register. On success it calls the login function from auth context and navigates to /dashboard. On failure it displays the error. If already authenticated, redirects to /dashboard.

**Dashboard page** calls GET /api/dashboard on mount. While loading it shows skeleton loaders in place of each data section. The page displays four sections: a Bac countdown widget at the top, a recent attempts list showing the last five completed quizzes with subject name, chapter name, score, and percentage, a bar chart using Recharts showing average percentage per chapter across all attempts, and a weak chapters section highlighting the three chapters with the lowest average percentage. The credit balance in the navbar updates from the returned creditBalance field. There is a quick-start button that navigates to /quiz.

**ExamLibrary page** calls GET /api/exams on mount with no filters, returning all exams. A filter bar at the top contains four dropdowns: subject, year, stream, and type. Changing any dropdown re-calls the API with the updated query parameters. Exams are displayed as cards showing the subject name, year, stream, type badge, and a button to open the PDF. The PDF opens in a new browser tab using the fileUrl directly — no download is involved. When no exams match the current filters, an empty state message is shown with a button to clear all filters. This page makes no authentication requirement — it is fully public.

**QuizSelect page** calls GET /api/subjects on mount to get all subjects and their chapters. Subjects are displayed as tabs or section headers, and chapters are listed under each subject. Each chapter card shows the chapter name and the credit cost prominently so the student knows what they are committing to before clicking. Clicking a chapter's Start Quiz button triggers the quiz start flow described in section 9.

**QuizActive page** receives the attemptId from the URL parameter. The questions array is passed via React Router's navigation state from QuizSelect — it is never re-fetched. If the state is missing for any reason, the page redirects to /quiz. Questions are displayed one at a time or all at once — this is a design decision, but showing all at once with a scroll is simpler and less error-prone. For MCQ questions, the answer inputs are radio buttons where each option is selectable. For numerical questions, the input is a number field. A progress indicator shows how many questions have been answered out of the total. A Submit button is active only when every question has been given at least an empty string answer — the backend handles missing answers as wrong. On submit, the page calls POST /api/quiz/submit and navigates to /quiz/:attemptId/results, passing the entire response object via router state.

**QuizResults page** receives the results data from React Router navigation state. It never makes an API call of its own. It displays the total score, the maximum score, the percentage, and a complete breakdown of every question: the question text, the student's answer, the correct answer, and whether it was right or wrong. Correct answers are shown with a green indicator and wrong answers with a red indicator. A Try Again button navigates back to /quiz. A Dashboard button navigates to /dashboard.

**Credits page** calls GET /api/credits/packs and GET /api/credits/history on mount. The packs are displayed as cards. Each card shows the pack name, the credit amount in large text, and the price in Algerian Dinars below it. A Buy button on each card triggers the purchase flow described in section 9. The transaction history is a list below the packs showing all past purchases with date, pack name, credits added, amount paid, and the status badge — pending, paid, or failed.

**PaymentSuccess page** calls GET /api/auth/me immediately on mount to fetch the updated user object with the new credit balance. It calls updateCredits from auth context with the returned balance. It displays a success message including the new credit balance and a button to go to /quiz to use the credits. If the /api/auth/me call fails, it still shows a success message but tells the student to refresh the page to see their updated balance.

**PaymentFail page** shows a failure message and offers two options: try purchasing again, which navigates back to /credits, and contact support. This page makes no API calls.

**AdminDashboard page** calls GET /api/admin/stats on mount. It displays four metric cards: total registered students, total questions in the database, total exams uploaded, and total revenue from paid transactions in Algerian Dinars. Below the metrics it shows the most attempted chapter. Navigation links to the other admin pages are prominent on this page.

**AdminExams page** calls GET /api/exams on mount with no filters to list all exams. The list shows each exam's subject, year, stream, type, and upload date with a Delete button on each row. An Upload Exam button opens a form. The form contains a subject dropdown populated from GET /api/subjects, a year number input, a stream text input, a type dropdown with exam and correction options, and a PDF file picker. On submit the form sends a multipart FormData POST to POST /api/admin/exams. An upload progress bar is shown while the file uploads. On success the new exam is prepended to the list in state without refetching. The Delete button opens a Shadcn AlertDialog that names the specific exam being deleted. On confirmation it calls DELETE /api/admin/exams/:id and removes the exam from the local list.

**AdminQuestions page** calls GET /api/admin/questions on mount with default filters. A filter bar allows filtering by subject, chapter, and question type. Questions are listed in a table showing the chapter name, question type, content truncated to 80 characters, and action buttons. An Add Question button opens a form. The form is conditional based on the selected type. For MCQ questions the form shows a content text area, between two and five option fields with an add option and remove option capability, a radio selector to mark which option is correct, and a points number input. For numerical questions the form shows a content text area, a correct answer number input, a tolerance float input with a helper explaining what tolerance means, and a points number input. The delete button on each question opens a Shadcn AlertDialog naming the question content. If the backend returns 409 because the question has existing answers, the AlertDialog is replaced by a toast explaining that the question cannot be deleted because students have already answered it.

**AdminCredits page** calls GET /api/admin/credit-packs on mount to list all packs including inactive ones. Each pack shows its name, credit amount, price, and active status with a toggle switch. The toggle calls PUT /api/admin/credit-packs/:id to flip the isActive value. An Add Pack button opens a form with fields for name, credits, priceDa, and an active toggle. On submit it calls POST /api/admin/credit-packs.

## 9. Key Interaction Flows

These flows have specific requirements that must be implemented exactly as described. Each one involves multiple steps and a specific data handoff between components.

**Credit purchase flow.** The student is on the Credits page viewing the available packs. They click Buy on a pack. The button immediately shows a loading spinner and is disabled to prevent double submission. The page calls POST /api/credits/checkout with the packId in the request body. The backend responds with a checkoutUrl string. The frontend immediately executes window.location.href = checkoutUrl, which sends the browser to Chargily's hosted payment page. The student completes payment on Chargily's page using their CCP or BaridiMob. Chargily redirects to /credits/success on success or /credits/fail on failure. The success page calls GET /api/auth/me to get the fresh user object with the updated credit balance. It calls updateCredits from auth context. The navbar updates immediately. The student sees a confirmation message with their new balance.

**Quiz start flow.** The student is on the QuizSelect page. They see chapters listed under each subject. Each chapter shows its credit cost. They click Start Quiz on a chapter. The button shows a loading spinner. The page calls POST /api/quiz/start with the chapterId. If the backend responds with 402, the Axios interceptor redirects to /credits — the QuizSelect page does not need to handle this case. On a 200 response, the data contains the attemptId and the questions array already stripped of correct answers. The page immediately navigates to /quiz/:attemptId, passing the questions array and the attemptId in React Router's navigation state. The updateCredits method is called with the creditBalance returned in the response. The QuizActive page reads the questions from navigation state and never makes another API call to fetch them.

**Quiz submit flow.** The student has answered all questions on the QuizActive page and clicks Submit. The button shows a loading spinner and is disabled. The page collects all answers into an array of objects, each with questionId and studentAnswer, and calls POST /api/quiz/submit with the attemptId and the answers array. On success the page navigates to /quiz/:attemptId/results, passing the entire response object in navigation state. The QuizResults page reads this data from navigation state and renders it without making any additional API call. If the student somehow lands on the results page without navigation state, they are redirected to /quiz.

**Chatbot message flow.** The ChatbotWidget is rendered on every authenticated page as a floating element in the bottom-right corner. Clicking it opens a slide-up chat panel. The student types a message and clicks Send. The input is cleared immediately and replaced with a typing indicator. The widget calls POST /api/chatbot with the message. If the backend responds with 402, the Axios interceptor redirects to /credits — the widget does not handle this individually. On success the response text is appended to the chat history stored in the widget's local state. The updateCredits method is called with the returned creditBalance. The credit balance in the navbar updates immediately. If the API fails for any other reason, a friendly error message is shown inside the chat panel. Chat history is stored in the widget's local component state only — it does not persist to localStorage and resets when the page is refreshed.

**Exam upload flow (admin).** The admin is on the AdminExams page and clicks Upload Exam. A form appears — either as a dialog or as an inline section. The form has fields for subject (populated from GET /api/subjects), year, stream, type, and a PDF file input. When the admin selects a file, the filename is displayed next to the input for confirmation. On submit, the form builds a FormData object — not a JSON object — appending each field and the file to it. It calls POST /api/admin/exams with the FormData as the request body. Axios detects the FormData automatically and sets the Content-Type to multipart/form-data. An upload progress bar is shown using Axios's onUploadProgress callback. On success the new exam is prepended to the exam list in state without refetching.

## 10. Components

**Navbar** is a shared component rendered on every page. When the user is not authenticated, it shows the platform logo on the left and Login and Register buttons on the right. When the user is authenticated as a student, it shows navigation links to the exam library, quiz, and dashboard, and on the right it shows the credit balance (formatted as a number followed by "crédits") and a logout button or user menu. When the user is an admin, it shows navigation links to the admin panel sections instead of the student links. The credit balance displayed comes from the creditBalance value in the auth context, so it updates in real time without any prop drilling.

**ProtectedRoute** is a wrapper component used in the route tree. It reads isLoading and isAuthenticated from auth context. While isLoading is true, it renders a centered loading spinner that occupies the full viewport. When isLoading is false and isAuthenticated is false, it redirects to /login, saving the current location in router state. When isAuthenticated is true, it renders the outlet or child component.

**AdminRoute** extends ProtectedRoute. After confirming the user is authenticated, it additionally checks whether user.role equals admin. If the user is authenticated but not an admin, it redirects to /dashboard silently.

**ChatbotWidget** is a floating component that appears on all authenticated pages. It renders a circular button fixed to the bottom-right corner of the viewport. Clicking it opens a slide-up panel anchored above the button. The panel contains a scrollable chat history area, a text input field at the bottom, and a Send button. The component manages its own open/closed state, its own chat history array in local state, and its own loading state for the in-flight API call. It imports the Axios instance and calls POST /api/chatbot directly. It reads updateCredits from useAuth to update the global credit balance after each successful message.

## 11. API Calls Reference

This section lists every API call the frontend makes, the component that makes it, when it is triggered, and what the frontend does with the response.

POST /api/auth/register is called by Register.jsx on form submission. On success the token and user from the response are passed to the login context function and the user is navigated to /dashboard.

POST /api/auth/login is called by Login.jsx on form submission. Same handling as register on success.

GET /api/auth/me is called by AuthContext on mount whenever a token exists in localStorage, and by PaymentSuccess.jsx on mount to refresh the credit balance after a successful payment.

GET /api/subjects is called by QuizSelect.jsx on mount to populate the subject and chapter list, and by the add question form in AdminQuestions.jsx to populate the subject dropdown.

GET /api/exams is called by ExamLibrary.jsx on mount and again whenever any filter value changes. The query parameters subjectId, year, stream, and type are appended when they have values.

GET /api/dashboard is called by Dashboard.jsx on mount. The entire response is destructured into the four dashboard data sections.

POST /api/quiz/start is called by QuizSelect.jsx when the student clicks Start Quiz on a chapter. On success the response data is passed to React Router's navigate function as state while navigating to the quiz page.

POST /api/quiz/submit is called by QuizActive.jsx when the student clicks Submit. On success the response data is passed as router state while navigating to the results page.

POST /api/chatbot is called by ChatbotWidget.jsx each time the student sends a message. The response text is appended to the local chat history.

GET /api/credits/packs is called by Credits.jsx on mount to list available purchase options.

POST /api/credits/checkout is called by Credits.jsx when the student clicks Buy on a pack. The checkoutUrl from the response is used in window.location.href to redirect.

GET /api/credits/history is called by Credits.jsx on mount to show past transactions.

GET /api/admin/stats is called by AdminDashboard.jsx on mount.

GET /api/admin/questions is called by AdminQuestions.jsx on mount and on filter change.

POST /api/admin/exams is called by AdminExams.jsx when the upload form is submitted with a FormData body.

DELETE /api/admin/exams/:id is called by AdminExams.jsx after the admin confirms the deletion dialog.

POST /api/admin/questions is called by AdminQuestions.jsx when the add question form is submitted.

PUT /api/admin/questions/:id is called by AdminQuestions.jsx when an existing question is edited.

DELETE /api/admin/questions/:id is called by AdminQuestions.jsx after the admin confirms the deletion dialog.

GET /api/admin/credit-packs is called by AdminCredits.jsx on mount.

POST /api/admin/credit-packs is called by AdminCredits.jsx when the add pack form is submitted.

PUT /api/admin/credit-packs/:id is called by AdminCredits.jsx when the admin toggles a pack's active status or edits its details.

## 12. UI Rules & Design Conventions

These rules apply to every page and component without exception. They are not suggestions.

Every button that triggers an API call must show a loading spinner while the request is in flight and must be disabled to prevent double submission. Never allow a form to be submitted twice. The loading state is a local boolean in the component that is set to true before the API call and false in the finally block.

Every page that loads data on mount must show skeleton loaders while the data is being fetched. Skeleton loaders are rectangular placeholder elements that match the approximate dimensions of the content they are replacing. They are available in Shadcn as the Skeleton component. Never show a blank white page while data loads.

Every list or data section must have a designed empty state. When a student has no quiz attempts, the recent attempts section shows a message encouraging them to start their first quiz with a button linking to /quiz. When no exams match the current filters, the exam list shows a message with a button to clear all filters. Empty states are never left as blank space.

The student's credit balance must always be visible in the navbar when authenticated. It must update in real time after every action that changes it — quiz start, chatbot message, and payment confirmation — without requiring any page reload or manual refresh.

Every delete action in the admin panel must use Shadcn's AlertDialog to ask for confirmation before the API call is made. The dialog title and description must name the specific item being deleted. A generic "are you sure?" message is not acceptable. The confirmation button uses the destructive variant.

The platform must be fully usable on mobile at a minimum viewport width of 375 pixels. Every page, form, and component must be tested at this width. The quiz question page specifically must be comfortable to read and answer on a phone. The chatbot bubble must not cover form inputs or primary action buttons on small screens.

The quiz page must never display any indication of whether an answer is correct or incorrect while the quiz is in progress. No green highlights, no checkmarks, no styling change on selection that implies correctness. The correct answers come from the backend only after submission.

Toast notifications are used for all success and error messages that need to be communicated without blocking the user interface. The Shadcn Toast component and useToast hook handle this. Success toasts are shown after successful form submissions, file uploads, and account actions. Error toasts are shown when an API call fails unexpectedly. Toasts auto-dismiss after four seconds. The browser's alert function is never used anywhere in the codebase.

## 13. Form Handling

Forms in this application do not use a dedicated form library like React Hook Form or Formik. The forms are simple enough to manage with controlled inputs and local state.

Every form input is a controlled component — its value is tied to a piece of local state, and its onChange handler updates that state. Submit handlers call preventDefault to stop the default browser form submission, run basic client-side validation, set a loading state to true, make the API call, handle success and error cases, and set loading state to false in a finally block.

Client-side validation is minimal and exists only to give immediate feedback for obvious mistakes — an empty required field, an invalid email format, or a password that is too short. It does not duplicate the backend's validation. If the backend returns a 400 with validation details, the frontend displays those messages next to the relevant fields.

The exam upload form is the one exception to the controlled input approach for files. File inputs cannot be controlled in the traditional React sense. The file is held in local state as a File object and appended to a FormData instance at submission time. The form itself does not use the JSON content type — it lets Axios detect the FormData and set multipart/form-data automatically.

## 14. Error Handling Strategy

There are four layers of error handling in the frontend.

The first layer is the Axios response interceptor. It handles 401 and 402 errors globally before any component sees them, as described in section 5.

The second layer is the component-level try/catch block. Every API call is wrapped in a try/catch. The catch block extracts the error message from err.response.data.error if available, or falls back to a generic message like "Something went wrong. Please try again." This message is either displayed as a toast notification or shown inline beneath the triggering form.

The third layer is empty state handling. When an API call succeeds but returns an empty array or null data, the component renders a designed empty state rather than a blank section.

The fourth layer is page-level error boundaries. React error boundaries catch rendering errors that slip through. A root-level error boundary in App.jsx shows a simple full-page error message with a button to reload the application.

What must never happen: a blank white page with no explanation, a component silently swallowing an error, or an unhandled promise rejection causing the app to freeze. Every failed API call must result in visible feedback to the user.

## 15. Deployment on Vercel

Vercel is used for the frontend deployment. The repository contains both the client and server folders. Vercel must be configured to only process the client subfolder.

When setting up the Vercel project, the root directory is set to client. The framework preset is Vite. The build command is npm run build. The output directory is dist. All three of these must be set explicitly even though Vercel can detect them automatically — explicit configuration prevents surprises if Vercel's auto-detection changes.

The environment variable VITE_API_URL is added in Vercel's project environment variables panel, pointing to the Render backend URL. It is not in any file that gets committed.

A vercel.json file must be present in the client folder. This file contains a single rewrite rule that maps all paths to /index.html. This is mandatory for any React single-page application deployed on Vercel. Without it, any user who refreshes a page other than the root — for example /dashboard, /quiz/42/results, or /admin/exams — receives a 404 error from Vercel because Vercel tries to find a static file at that path and finds nothing. The rewrite rule tells Vercel to always serve index.html and let React Router handle the routing client-side.

Every push to the main branch triggers an automatic production deployment on Vercel. Every push to any other branch creates a preview deployment at a unique URL. Preview deployments are useful for testing a feature branch before it is merged into main.

The frontend cold start time on Vercel is essentially zero — Vercel's CDN serves the static files immediately. The perceived slowness on first load comes from the Render backend waking up from its free tier sleep state, not from the Vercel deployment.
